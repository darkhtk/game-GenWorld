## 프레임워크 개요

4개의 Claude Code CLI 에이전트가 2분 간격 루프로 병렬 동작하는 파일 기반 오케스트레이션:

| 에이전트 | 역할 | 루프 수 |
|----------|------|---------|
| **Supervisor** | 코드 감사, 에셋 생성, 버그 수정 | 36루프 |
| **Developer** | 태스크 구현, 테스트 작성 | - |
| **Client** | 멀티 페르소나 코드 리뷰 (4가지 관점) | - |
| **Coordinator** | BOARD 동기화, RESERVE 보충, SPEC 선제 작성 | - |

## 정성 평가

### 이 프레임워크는 무엇에 강한가

1. **압도적 처리량**: 2일간 139태스크 / 475커밋 / 94클래스 감사. 사람이 수주 걸릴 작업을 자동 수행.
2. **자기 복구 루프**: NEEDS_WORK 4건이 전부 v2에서 자동 해결. Rejected→Rework→Re-review 사이클이 사람 개입 없이 동작.
3. **리뷰 깊이**: Client 에이전트의 4가지 검증(엔진/코드/UI/플레이 시나리오) x 4가지 페르소나 리뷰가 실질적 품질 게이트로 기능. REVIEW-S001-v1에서 SHA256 체크섬 체인 분석, 7가지 플레이 시나리오 추적까지 수행.
4. **역할 분리 설계**: Supervisor(감사/에셋) / Developer(구현) / Client(검증) / Coordinator(동기화)의 관심사 분리가 명확하고, 실제로 각 역할이 독립적으로 기능함.

### 무엇이 약한가

1. **관측성 부재**: 토큰 사용량, 비용, 에이전트 성공/실패율, 응답 시간 등 운영 지표 수집 구조 없음.
2. **권한 강제력 미흡**: 에이전트 권한이 프롬프트 지시에만 의존. 파일시스템 레벨 제한 없어 에이전트 hallucination 시 월권 가능.
3. **비용 불투명**: 2분 x 4에이전트 루프가 단순 "이미 구현됨 확인"에도 전체 사이클을 돌아 낭비 발생. 비용 상한선 제어 불가.
4. **워크플로우 경직성**: 2분 간격 폴링 방식이 유일한 트리거. In Review 등록 시에만 Client를 기동하는 등의 이벤트 기반 실행 불가.

### 어떤 조건에서 적합한가

- 개인/소규모 팀의 **빠른 프로토타이핑 및 코드 안정화**
- LLM 비용에 제약이 적은 **실험/연구 환경**
- **게임 개발** 등 에셋+코드+리뷰가 동시에 돌아야 하는 도메인
- 사람의 지속적 모니터링이 가능한 **단기 집중 작업** (수일 이내)

### 어떤 용도로는 부적합한가

- 수십 명 규모의 팀에서 **엔터프라이즈 레벨 CI/CD 파이프라인** 대체
- **비용 민감한 운영 환경** (API 호출량 제한 필요)
- **장기 무인 운영** (모니터링/알림/자동 복구 인프라 부재)
- **규제 환경** (감사 로그 표준, 접근 제어, 컴플라이언스 대응 미비)

### 도입 시 핵심 리스크

1. **API 비용 폭증**: 2,880회/일 x Opus급 모델. 비용 상한 메커니즘 없음.
2. **git 충돌**: 4에이전트가 동시에 같은 파일(BOARD.md) 수정 시 충돌 가능. `git pull --rebase` 실패 시 대응 불명확.
3. **권한 위반**: 프롬프트 기반 권한 제어는 LLM의 지시 준수율에 의존. 복잡한 컨텍스트에서 월권 가능성.
