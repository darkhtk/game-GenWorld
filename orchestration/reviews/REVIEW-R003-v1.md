### 코드 품질 소견

**강점:**
- `version ?? 0` — 간결하고 정확한 레거시 감지
- v0 세이브는 envelope 없이 전체 JSON이 데이터 → root를 그대로 사용하는 로직이 자연스러움
- 마이그레이션 전 원본 JSON 문자열로 백업 (`CreateBackup(json)`) — 바이너리 손실 없음
- 누락된 마이그레이션 단계는 Warning만 (크래시 아님) — 안전한 fallthrough
- try/catch로 전체 로드 실패 시에도 백업 생성

**참고:**
- `SaveMigrations`와 `SaveSystem`이 같은 파일에 있음 — 코딩 표준 "파일당 하나의 클래스" 위반이나, 밀접한 관계로 실용적 판단. 향후 마이그레이션이 늘어나면 분리 권장.

## 페르소나 리뷰

### 🎮 캐주얼 게이머
세이브 날아가는 게 제일 싫은데, 업데이트해도 내 세이브 유지된다니 좋다. 실패해도 백업 만들어준다니까 안심.

### ⚔️ 코어 게이머
마이그레이션 체계가 확장성 있다. v0→v1은 빈 변환이지만 향후 스킬 리워크나 스탯 재설계 시 v1→v2 변환 함수만 추가하면 된다. 다단계 순차 적용 설계로 버전 간 점프도 대응.

### 🎨 UX/UI 디자이너
마이그레이션 성공 시 토스트 알림이 있으면 좋겠지만 optional로 명시되어 있으니 OK. 실패 시 "새 게임 시작" 안내 UI는 별도 구현 필요 — 현재는 null 반환만.

### 🔍 QA 엔지니어
- 백업 경로가 `.bak` 단일 파일 → 연속 마이그레이션 실패 시 이전 백업 덮어씀. 치명적이진 않으나 타임스탬프 백업 고려 가능.
- `File.WriteAllText` / `File.ReadAllText` 사용 — 대용량 세이브에서 메모리 이슈 가능하나 RPG 세이브 크기로는 충분.
- `Apply()`에서 migration 함수가 null 반환하면 이후 단계에서 NRE 발생 가능 — 현재 MigrateV0ToV1은 data를 그대로 반환하므로 안전하지만, 향후 마이그레이션 작성 시 주의 필요.
