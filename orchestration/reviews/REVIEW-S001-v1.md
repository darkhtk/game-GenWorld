## 검증 2: 코드 추적

| 항목 | 결과 | 비고 |
|------|------|------|
| TASK 명세 부합 | ✅ | SHA256 체크섬 + 백업 로테이션(3) + 자동 복원 — 보드 설명과 일치 |
| 기존 코드 호환 | ✅ | API 시그니처 동일 (Save/Load/HasSave/DeleteSave), 호출부 변경 불필요 |
| 아키텍처 패턴 | ✅ | static 유틸 클래스, CLAUDE.md "Systems: 순수 C#" 준수 |
| 테스트 커버리지 | ⚠️ | 기존 roundtrip 테스트 6건 통과하나, 신규 기능(체크섬 검증/백업 복구/로테이션) 전용 테스트 미작성 |

**코드 품질 소견 (깊은 리뷰):**

1. **체크섬 무결성 체인:** `JObject.FromObject → ToString(Formatting.None) → SHA256` 순서로 직렬화 결정론적. 저장 시와 로드 시 동일한 `Formatting.None` 사용 — 일관성 보장됨.
2. **백업 로테이션 안전성:** `RotateBackups()`는 `File.Copy(SavePath, bak1, true)` 사용 — 원본 삭제 없이 복사. WriteAllText 실패 시에도 기존 세이브 + 백업 모두 보존됨.
3. **v0 하위호환:** checksum 필드 없는 레거시 세이브는 `storedChecksum != null` 분기로 검증 스킵 — 기존 유저 세이브 깨지지 않음.
4. **DeleteSave() 확장:** 백업 파일도 함께 삭제 — "새 게임" 시 고아 백업 잔류 방지.
5. **에러 핸들링:** Save/TryLoadFrom/RotateBackups 각각 독립 try-catch. 백업 실패가 세이브를 차단하지 않음.
6. **잠재 이슈 — 낮음:** 파일 락 없음. 단일 플레이어 게임에서 동시 접근 가능성 극히 낮아 현 단계에서 수용 가능.

## 페르소나 리뷰

### 🎮 캐주얼 게이머

저장 눌렀다가 게임 꺼졌을 때 세이브가 날아가면 진짜 화나는데, 백업이 3개나 있으면 안심이다. 근데 "백업에서 복구했어요" 같은 메시지가 화면에 나오면 더 좋겠다. 콘솔 로그는 일반 플레이어가 못 보니까.

### ⚔️ 코어 게이머

체크섬으로 변조 감지하는 건 좋은데, 이게 치트 방지 목적이라면 SHA256만으로는 부족하다 — 키 없이 해시만 저장하면 데이터+해시 동시 수정 가능. 하지만 세이브 **무결성 보호**(크래시/손상 감지)가 주 목적이라면 충분하다. 백업 로테이션 3개는 적절한 수치. 단, 자동 저장 빈도가 높으면 3개가 금방 돌아가니 MaxBackups를 설정에서 조정 가능하게 해도 좋겠다.

### 🎨 UX/UI 디자이너

현재 복구 과정이 완전히 투명(사용자 인지 불가). UX 관점에서:
- 복구 성공 시 비침투적 토스트 알림 권장 ("이전 저장에서 복원되었습니다")
- 전체 복구 실패 시 "새 게임을 시작하시겠습니까?" 대화상자 유도 필요
- 현 구현에서 Load() null 반환 → GameManager.cs:852에서 처리 확인 필요

단, 이번 태스크 스코프는 **시스템 레벨** 복구이므로 UI 연동은 후속 태스크로 적절.
