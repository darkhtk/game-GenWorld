## 페르소나 리뷰

## 🎮 Alex (캐주얼 게이머)
### 인상
이제 버프 스팸이 막혔네! 전에는 rage 계속 써서 무한 버프 받을 수 있었는데, 이제 3번만 중첩되고 끝. 근데 이게 좋은 건지는 모르겠음.
### 문제점
- 버프 중첩 횟수가 게임에서 어디에도 표시 안 됨. 몇 번 쌓였는지 알 수가 없어서 답답함
- 3중첩 제한이 있다는 걸 게임에서 알려주지 않음
### 제안
- 버프 아이콘에 "x3" 같은 숫자 표시 추가
- 최대 중첩 도달했을 때 약간의 시각적 피드백

## ⚔️ Marcus (코어 게이머)  
### 인상
시스템적으로 깔끔한 개선. 무한 스택킹 방지는 필수였고, 타입별 차별화도 합리적. rage 3중첩은 적당한 밸런스 포인트.
### 문제점
- 스택 상한이 하드코딩되어 있음. 향후 밸런스 조정 시 코드 수정 필요
- dot/slow의 value 갱신 로직과 stack count가 따로 놈. value는 최대/최소로 갱신되지만 stack은 계속 증가
### 제안
- 향후 data-driven 방식으로 개선 (ScriptableObject 등)
- 스택별 추가 보너스 시스템 고려해볼 만함

## 🎨 Sarah (UX/UI 디자이너)
### 인상
백엔드 구조는 잘 만들어졌지만, 사용자에게는 전혀 보이지 않는 개선. 정보 표시가 없으면 의미가 반감됨.
### 문제점
- 스택 정보가 UI에 노출되지 않음. ActiveEffectInfo에 필드는 추가했지만 실제 표시하는 곳이 없음
- 스택 제한에 대한 사용자 피드백 부재
- 일관성 없는 스택 정책 (heal은 무제한, 나머지는 제한)
### 제안
- HUD 버프 아이콘에 스택 카운터 표시
- 최대 스택 도달 시 subtle한 시각적 표시
- 일관된 스택 정책 가이드라인 수립

## 🔍 David (QA 엔지니어)
### 인상
구현은 견고하고 테스트 커버리지도 충분. 모든 edge case가 고려되었고 하위 호환성도 완벽.
### 문제점
- stun/slow에서 stack count 증가 조건이 애매함. maxStack 체크는 하지만 value 갱신 후에 stack이 증가함
- ApplyDot() 메서드에서는 stack count를 고려하지 않음 (value만 비교)
### 제안
- stun/slow의 stack 증가 조건을 명확히 정의
- ApplyDot에도 stack cap 로직 적용 고려
- Integration test 추가 (실제 게임 플로우에서의 동작 검증)
