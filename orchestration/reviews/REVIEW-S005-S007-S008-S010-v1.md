## S-005: InventorySystem OccupiedSlots LINQ 할당 제거

**스펙:** SPEC-S-005.md | **방식:** Option A (수동 카운트 루프)

### 변경 내용 (InventorySystem.cs:7-16)

```csharp
// Before: public int OccupiedSlots => Slots.Count(s => s != null);
// After:
public int OccupiedSlots
{
    get
    {
        int count = 0;
        for (int i = 0; i < Slots.Length; i++)
            if (Slots[i] != null) count++;
        return count;
    }
}
```

### 검증

| 항목 | 결과 | 비고 |
|------|------|------|
| LINQ Count() 제거 | ✅ | for 루프로 완전 교체 |
| 동일 반환값 | ✅ | null이 아닌 슬롯 개수 — 로직 동일 |
| GC 할당 제거 | ✅ | IEnumerator/closure 할당 없음 |
| 기존 코드 호환 | ✅ | public property 시그니처 변경 없음 |

**AC 충족:** 4/4 ✅

## S-007: CombatManager stale ref 방어

**스펙:** SPEC-S-007.md | **방식:** 분산형 null/dead 가드

### 변경 분석

스펙의 "리스트 필터링" 대신, **사용 지점마다 null/dead 가드**를 적용하는 방식:

| 위치 | 가드 | 비고 |
|------|------|------|
| PerformAutoAttack loop (line 65) | `if (m == null \|\| m.IsDead) continue` | 이터레이션 스킵 |
| HandleMonsterAttacks loop (line 103) | `if (m == null \|\| m.IsDead) continue` | 이터레이션 스킵 |
| Projectile arrival (line 139) | `if (m == null \|\| m.IsDead) return` | 콜백 조기 반환 |
| DealDamageToMonster (line 327) | `if (m == null \|\| m.IsDead) return false` | 데미지 시도 차단 |
| ExecuteSkill (line 223) | `if (monsters == null) return` | null 리스트 early return |

### 아키텍처 참고

`_cachedMonsters`는 `monsterSpawner.ActiveMonsters`의 참조(복사 아님). `RemoveMonster()`가 리스트를 수정하면 캐시된 참조도 반영됨. 따라서 "stale list" 이슈는 본질적으로 경미하나, `Destroy(obj, 0.5f)` 딜레이 동안 dead 참조가 잔존할 수 있으므로 `m.IsDead` 가드가 필수.

### 검증

| 항목 | 결과 | 비고 |
|------|------|------|
| NRE 방지 | ✅ | 모든 사용 지점에서 null 체크 |
| dead 몬스터 타겟 방지 | ✅ | IsDead 가드로 스킵 |
| 기존 호환 | ✅ | 가드만 추가, 로직 변경 없음 |
| 할당 없음 | ✅ | 새 리스트 생성 없음 |

**AC 충족:** 4/4 ✅

## 페르소나 리뷰

### 🎮 캐주얼 게이머

이 수정들은 다 보이지 않는 곳에서 일어나는 방어 코드다. 플레이어 입장에서는 "갑자기 튕기는 일이 줄어들겠다" 정도. 특히 S-010 퀘스트 보상 null 방어가 마음에 든다 — 퀘스트 깨고 보상이 안 뜨는 건 기분이 정말 나쁜데, 이제 최소한 에러 없이 넘어간다.

### ⚔️ 코어 게이머

4개 모두 런타임 안정성 개선. 성능 관점:
- **S-005:** LINQ → for 변환은 교과서적 최적화. 인벤토리가 매 프레임 갱신되면 체감 가능.
- **S-007:** 분산 가드 방식은 리스트 필터링보다 할당이 없어 더 가볍다. 다만 가드 누락 리스크가 있으므로 새 사용처 추가 시 주의 필요.
- **S-008:** Camera.main 캐싱은 모바일/저사양에서 차이가 난다. Unity 표준 패턴 준수.
- **S-010:** null coalescing으로 간결하게 처리. 깔끔.

### 🎨 UX/UI 디자이너

S-008이 UX에 가장 직결. 카메라 null 시 마우스 에이밍이 동작 안 하면 플레이어가 조작 불능 상태에 빠진다. 재획득 로직으로 씬 전환 후에도 즉시 조작 복구 — 좋다.
